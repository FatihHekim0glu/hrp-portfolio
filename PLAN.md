## Project #9 — Hierarchical Risk Parity (HRP)

### Objective & honest headline
Implement Lopez de Prado's HRP (2016) from scratch — tree clustering, quasi-diagonalization, recursive bisection — as a pure, typed compute library, and benchmark it rigorously out-of-sample against three honest baselines: Markowitz (min-variance + max-Sharpe), inverse-variance (IVP), and the DeMiguel 1/N naive portfolio. The deliverable answers ONE question correctly: does HRP deliver what the paper claims (lower OOS variance, no singular-matrix blowups, lower concentration) and does that translate into a STATISTICALLY SIGNIFICANT risk-adjusted-return edge over naive diversification once realistic transaction costs and estimation error are paid for?

**Honest headline (locked as a regression):** HRP does what it advertises on the dimension it actually targets — materially LOWER out-of-sample portfolio variance than Markowitz CLA, and it never blows up on near-singular covariance — but it does NOT reliably beat 1/N on OOS Sharpe after costs. On a PIT S&P 500 universe with walk-forward rebalancing and ~10 bps/side costs, the HRP-vs-1/N Sharpe gap is economically small and statistically insignificant under Jobson-Korkie-Memmel, with the Deflated Sharpe near zero once the FULL configuration grid is counted. This is the literature-consistent outcome (de Prado 2016 claims lower OOS variance, not higher Sharpe; DeMiguel-Garlappi-Uppal 2009 makes 1/N a brutal benchmark): HRP is a better-behaved RISK allocator, not a return generator. Its lower turnover is its most defensible practical edge over Markowitz, and the README frames the insignificant Sharpe gap as the honest finding.

### Method
From-scratch HRP exactly per de Prado, then a walk-forward horse race against IVP, Markowitz (min-var + max-Sharpe), and 1/N. Three hand-rolled, parity-tested stages:
1. **Tree clustering** — correlation distance `d_ij = sqrt(0.5*(1-rho_ij))` (HONESTY: not `1-rho`), then a second-order Euclidean distance over the distance columns, then scipy single-linkage agglomerative clustering (paper default; `ward`/`complete`/`average` exposed as configurable ablations only).
2. **Quasi-diagonalization** — `getQuasiDiag` recovers the dendrogram leaf order so the reordered covariance places large covariances along the diagonal.
3. **Recursive bisection** — `getRecBipart` top-down split, allocating between clusters inversely to cluster variance, where cluster variance uses INVERSE-VARIANCE intra-cluster weights via `getClusterVar` (HONESTY: not equal weights).

**Shared-estimator fairness:** all four allocators are fed the IDENTICAL covariance estimator on each window (Ledoit-Wolf shrinkage by default, optional Marchenko-Pastur RMT clip), isolating the ALLOCATION RULE as the only treatment. Because max-Sharpe additionally needs expected returns, its `mu` is an explicit James-Stein / grand-mean-shrunk estimator (ADR-documented), with naive-sample-`mu` reported as an ablation — so the reader sees that it is `mu`-estimation noise, not the allocator, that sinks max-Sharpe. The HEADLINE comparison (HRP-vs-1/N) is covariance-only and `mu`-immune.

**Significance:** Jobson-Korkie-Memmel Sharpe-difference test + Politis-Romano stationary block-bootstrap CIs on the Sharpe gap + Deflated Sharpe Ratio with the FULL configuration grid as n_trials.

### Data strategy & survivorship
- **Sources:** Polygon.io Starter ($29/mo) grouped-daily bars as the survivorship-aware PIT path via the shared `api/lib/polygon/provider.py` `make_provider()`/`get_eod()`; `yfinance` (curl_cffi Chrome impersonation) -> Stooq fallback as the local/dev default (reuse risk-metrics/markowitz `data.py`), parquet/diskcache cached.
- **PIT universe:** rebuilt at each rebalance from constituents KNOWN at that date via the shared `markowitz/pairs data_providers/sp500_universe.py` (Polygon grouped-daily intersection). Delisted names stay in the panel through delisting (final return included), then drop out.
- **Survivorship (HIGHLY RELEVANT):** a static current-S&P-500 universe would inflate every allocator's return and could spuriously favor concentration. Mitigation is the PIT builder above. **Residual blind spot (README Limitations):** Polygon Starter has no true historical index-membership snapshots, so the PIT universe is a liquidity/price-traded approximation, and delisting-return completeness is imperfect — named explicitly.
- **New-constituent rule (explicit):** a ticker entering the universe at rebalance `t` with fewer than `lookback_window` in-sample observations known strictly before `t` is EXCLUDED from that window's allocation (never padded), and the exclusion is logged. Documented as the "estimable-universe" caveat (a longer-listing bias, distinct from survivorship, disclosed not hidden).
- **Risk-free:** PIT FRED-via-CSV risk-free (reuse risk-metrics/factorlab `data.py`), aligned and shift-safe; consumed by Sharpe/JKM.
- **Cost note:** Polygon gives Markowitz CLEAN PIT split/dividend-adjusted data so the horse race is not rigged by a dirty feed; yfinance/Stooq is free and sufficient for dev/CI. Assume NO server-side Supabase persistence today; rely on local parquet/diskcache.

### Leakage & correctness guards
- Covariance, Ledoit-Wolf shrinkage, the Marchenko-Pastur edge `q=T/N`, the dendrogram, AND all weights are estimated on the in-sample window ONLY; weights apply to the SUBSEQUENT OOS window via `signal.shift(1)` at the rebalance boundary.
- **Estimator-level no-lookahead test:** `shrinkage_intensity` and the RMT cutoff are asserted to be pure deterministic functions of the in-sample window (future-perturbation-invariance extended below the final weights).
- Returns via `pct_change(fill_method=None)` — never ffill prices before differencing.
- PIT universe selection uses only membership/liquidity known at that date (no forward-filled constituent list).
- Purge+embargo re-derived for the PORTFOLIO setting in an ADR: with non-overlapping daily returns and `shift(1)` application, embargo = return-horizon (=1 day daily); purge removes the single shared boundary observation. (Cargo-culting the pairs config is explicitly rejected.)
- **Parity oracle:** hand-rolled `getQuasiDiag`/`getRecBipart`/HRP weights validated to 1e-7 against PyPortfolioOpt `HierarchicalRiskParity` AND a second reference (mlfinlab/Riskfolio-Lib), reconciling tie-breaking in `getQuasiDiag` and inverse-variance vs equal in `getClusterVar`.
- **Singular-covariance regression test:** a block-perfectly-correlated matrix on which Markowitz CLA fails to invert MUST still yield valid HRP weights (the paper's headline robustness claim, encoded).
- **n_trials honesty:** DSR `n_effective_trials` counts the FULL explored grid — #allocators x #linkages x #covariance-estimators x #rmt(on/off) x #rebalance-freqs x #cost-levels x #lookback-windows — using the full `(k+2)/4` kurtosis term.
- **headline_verdict** is a PURE FUNCTION of `(jkm_pvalue, deflated_sharpe, bootstrap-CI sign)` from a fixed enum — it cannot emit "HRP beats 1/N" while the CI straddles zero (truth-table unit-tested).
- HAC/Newey-West (Andrews 1991 bandwidth) on any mean-return/alpha inference; never a single-window point estimate.
- Inline HONESTY-REQUIREMENT comments at each footgun (distance formula; `getClusterVar` weighting; single vs ward default drift; full vs excess kurtosis; raw vs shrunk `mu`).

### Risks & pitfalls
- Rigged-comparison risk (different estimators per allocator, OR naive-`mu` sinking max-Sharpe) — mitigated by shared-covariance + shared-shrunk-`mu` ADR + tests; headline kept on the `mu`-immune HRP-vs-1/N axis.
- Linkage drift (silent ward/average default) — single is the validated default, others ablations; parity test catches drift.
- `getClusterVar` equal-vs-inverse-variance footgun and `1-rho` vs `sqrt(0.5*(1-rho))` footgun — golden tests.
- Survivorship + new-constituent biases — PIT builder + exclusion rule + documented Polygon-membership blind spot.
- 1/N is a brutal benchmark (DeMiguel 2009): HRP expected to LOSE/tie on Sharpe — resist cherry-picking; report bootstrap CI + DSR straddling zero; verdict is pure-derived.
- Multiplicity: FULL configuration grid counted in DSR n_trials.
- Cold-start/512MB: block-bootstrap resample budget (not clustering) is the OOM/latency risk; cap n_assets/n_rebalances/n_bootstrap, lazy-import cvxpy, vendor [data] not [all], gate large grids behind 422/async.
- Import-purity: subprocess test guards vendoring.
